Why IRIS Stands Out
🕌
Cultural Intelligence
Adhan prayer time muting, Ramadan scheduling rules, RTL Arabic UI, GCC-compliant VAT invoicing with TRN per entity.
🌐
Multi-Region Architecture
Data stays in-country. Separate AWS stacks for UAE, Saudi, Bahrain, and US. Full PDPL and local data residency compliance.
🎙
AI Voice Interface
AWS Transcribe + Claude (Bedrock) + AWS Polly. Bounded intent processing — only pod-related commands are executed.
📡
Resilient IoT
Raspberry Pi runs fully offline during network drops. Local state machine maintains session, disinfection, and wake/sleep cycles.
🔒
Zero-Trust Hardware
Mobile app reads Device Shadow directly but writes ONLY through Backend API. All commands authenticated and ordered via FIFO SQS.
⚡
Smart Disinfection
Automated UV blue-light and ionization cycle after each session. Presence sensor confirms pod is empty before cycle starts.

Key System Flows

Step-by-step walk-through of the most critical data and control paths in IRIS.

1
📱
Customer selects pod & time slot
User opens Mobile App, browses pod availability, selects a slot, and proceeds to payment. The app sends the booking request via HTTPS through CloudFront.
Mobile AppCloudFront
2
🔐
Authentication & authorisation
API Gateway validates the JWT token with Cognito. Route 53 routes the request to the correct regional AWS stack (UAE, Saudi, etc.) based on geographic DNS.
Route 53API GatewayCognito
3
📅
Booking Service checks availability & cultural rules
Booking Service validates slot availability, applies Adhan-aware scheduling (no sessions during prayer times if configured), and checks Ramadan schedule rules via Regional Config Service.
Booking ServiceRegional ConfigAdhan API
4
💳
Payment processed via Checkout.com
Payment Service initiates a tokenized charge through Checkout.com. Supports Mada, Benefit Pay, Apple Pay, and Google Pay. Chargebee handles subscription metering and VAT invoice generation with GCC TRN.
Payment ServiceCheckout.comChargebee
5
🔄
Session State Machine created → Pod pre-warm scheduled
Session Service creates a state machine entry (booked state). Pod Lifecycle Manager schedules a wake sequence via EventBridge to trigger exactly T-2 minutes before the session start.
Session SMPod Lifecycle ManagerEventBridge
6
📡
Wake command sent to pod via IoT Core
EventBridge triggers an MQTT message to the pod's Raspberry Pi via AWS IoT Core MQTT Broker. Greengrass agent on the Pi receives the desired state update and begins the wake sequence.
MQTT BrokerDevice ShadowGreengrass
7
🌡️
Pod hardware initialises
Device Manager translates IRIS commands into Electric Set protocol. HVAC sets temperature, Privacy Glass clears, Availability Halo turns amber (pre-warm). Access lock primes for customer's arrival.
Device ManagerElectric SetHVAC · Glass · Lock
8
🔓
Customer arrives → Pod unlocks → Session goes active
Customer taps app or NFC at pod. Access Control unlocks. Presence sensor confirms occupancy. Session State Machine transitions to active. Real-time state updates flow to Mobile App via Device Shadow.
Session ActivePresence → IoT RulesMobile App UI updates
1
🗣️
User speaks to IRIS
Customer speaks a command in English or Arabic. The Mobile App or Kiosk App begins streaming the audio in real-time.
Mobile AppKiosk App
2
🎤
AWS Transcribe converts speech to text
Streaming STT with automatic noise threshold. Supports EN, AR, ES. Confidence scoring applied — if confidence is below threshold, system falls back to touch mode UI. Audio is stored in-region S3 bucket for audit.
AWS TranscribeS3 (in-region)
3
🎯
Intent Router classifies the request
The transcript text is sent to the Intent Router. It performs initial NLU dispatch and bounds the action space. Off-topic queries (outside pod control) are filtered here before reaching Bedrock — reducing cost and preventing misuse.
Intent Router
4
🧠
AWS Bedrock (Claude) processes bounded intent
The transcript plus session context (pod state, user preferences, current session) is sent to Claude on AWS Bedrock. Claude classifies the intent and returns a structured action object. Off-topic requests receive predefined H1/H2 redirects.
AWS Bedrock · Claude
5
📡
Device command issued to pod
The structured action (e.g. "set temperature to 21°C") is sent by Device Command Service to SQS FIFO queue. The command flows via EventBridge → MQTT → Raspberry Pi → Electric Set → Physical Device.
Device Command ServiceSQS FIFOMQTT
6
🔊
AWS Polly delivers spoken response
Bedrock's natural language response is sent to AWS Polly for TTS synthesis. Voice is EN-ZA (South African female). Whisper mode is automatically applied for privacy pod types. Audio streams back to the app.
AWS PollyMobile / Kiosk App
1
🚪
Session ends — grace period begins
Session Service detects session timeout or manual checkout. State Machine transitions: active → grace period. Customer has a configurable window to gather belongings. Notification sent to mobile app.
Session SMNotification Service
2
👁️
Presence sensor confirms pod is empty
CO₂ and presence sensors (Bevenic) continuously report telemetry via MQTT → IoT Rules Engine → DynamoDB. Rules Engine detects occupancy=false and triggers Presence event → Session State Machine.
Bevenic SensorsIoT Rules Engine
3
🧹
Disinfection State Machine activates
Trigger condition met: session_closed AND presence=empty. DSM initiates the disinfection cycle. If a motion event re-triggers (someone enters), the cycle is safely delayed. Privacy Glass goes opaque.
Disinfection SMSonte Glass → Opaque
4
🔵
Blue-light + ionization cycle runs
Commands flow via MQTT → Raspberry Pi → Electric Set → Baumedi unit. UV blue-light activates for the configured duration. Ionization unit runs in parallel. HVAC continues running to clear air.
Raspberry PiBaumedi · HVAC
5
✅
Pod resets → Ready for next booking
Disinfection cycle completes. Pod Lifecycle Manager resets all device states to defaults. Availability Halo turns green. Device Shadow updated. Pod is available for next booking immediately.
Pod Lifecycle ManagerDevice Shadow updatedHalo → Green
1
🌡️
Sensor data measured by Bevenic (temp / CO₂)
Bevenic smart device continuously measures temperature and CO₂ levels inside the pod. Readings are sent to the Electric Set control board which relays them to the Device Manager on the Raspberry Pi.
BevenicElectric SetDevice Manager
2
📤
Telemetry published via Greengrass → IoT Core
Device Manager publishes the telemetry to the local Greengrass agent. Greengrass relays it over MQTT TLS to AWS IoT Core using the topic pattern: vel/{region}/{site}/{cluster}/{pod}/{device}. QoS 1 ensures at-least-once delivery.
Greengrass v2MQTT · TLS 1.2+ · QoS 1
3
⚙️
IoT Rules Engine routes telemetry
Rules Engine applies SQL-like rules: telemetry snapshots go to DynamoDB, anomaly alerts go to SNS (which triggers Ops Dashboard notifications), presence events trigger the Session State Machine transition.
IoT Rules EngineDynamoDBSNS alerts
4
🪞
Device Shadow updated with reported state
The Raspberry Pi reports its current state to the Device Shadow (reported state). The Shadow stores both desired (what cloud wants) and reported (what device has confirmed) states. Delta sync happens automatically on reconnect.
Device ShadowDevice Registry
5
📱
Mobile App reads Shadow directly for real-time UI
The Mobile App uses the AWS IoT SDK to read the Device Shadow directly (not via the backend) for real-time sensor display — temperature, CO₂ level, light state. This reduces API latency and backend load for UI updates.
Mobile AppAWS IoT SDK (read-only)
